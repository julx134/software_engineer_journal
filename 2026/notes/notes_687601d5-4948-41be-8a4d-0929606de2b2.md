- As part of our weekly meeting, we usually update our manager on what we've accomplished and discuss any remaining tasks. For this particular meeting, my updates were approved, and I was assigned two features to complete before our big deployment in late Spring 2026. The first feature is to auto-assign IoT buttons to each room, and the second is to implement a Redis cache that populates when the server starts. Usually, these meetings take about 30 minutes and are straightforward, but this conversation ended up running much longer. The requirements were a bit unclear, which reminds me of why it's so important to clarify requirements before jumping into a solution. I initially had zero context, and my manager gave me only a sentence or two for each feature. Before leaving the meeting, I took the initiative to clarify the specifics.

- The first task was auto-assigning buttons to each room. This was vague because he simply said that once the CRUD features for the IoT button entities were finished, I should take over and auto-assign them. Specifically, each room requires three buttons: two buttons of type C-10 or B-10, and one "pull" button. I needed to understand exactly what "auto-assignment" entailed. My first thought was to clarify the exact button composition per room. I asked whether each room needed two of each (B-10 and C-10) or two total from that set. I also questioned if we would ever use only one specific type. My manager pulled up a diagram showing that each room requires a total of three buttons: one pull button and two of a single type (either B-10 or C-10). The reason it could be either type was simply that he hadn't confirmed which one we had in stock.

- Beyond the composition, I had to clarify the logic for the assignment process itself. I asked if we should roll back the entire process if we didn't have enough buttons to fulfill all rooms. For instance, if we had 20 rooms but only 10 buttons. He clarified that we wouldn't roll back; we would keep whatever assignments were successfully made. My next question was about partial fulfillment: if a room requires three buttons but we only have one or two available, do we still assign them or skip that room? He confirmed that we do not allow partial fulfillment; we only perform full fulfillments per room. With those details settled, the algorithm itself is simple: I just need to identify rooms without assigned buttons and allocate them accordingly.

- The second feature was more difficult to scope because it was tightly coupled with business logic. He originally stated that on server startup, the Redis cache should look for a list of events, and if five or more of a particular type exist, we should send a WebSocket event to the frontend client. I had to clarify if we even had a service to read from Redis during application startup. My manager noted we didn't, but suggested I could implement it if time allowed. I was still confused and tried to summarize his requirement, but my interpretation was incorrect. He then shared his screen to show me the actual requirement: on application startup, a function or script should read from the database to check for unacknowledged events. Any unacknowledged events should be cached into the Redis database. If Redis then contains enough events of a particular type, we trigger the WebSocket event to the frontend.

- For clarifications like these, I’ve learned to always write a sentence or two that explains the entire task in my own words and repeat it back to my manager. If my interpretation is correct, he approves it; if not, we refine it until the ambiguities are gone. This is a process I've developed through experience. One thing I love about software engineering and something I’ve learned about my own workflow is that you should never start implementing big system designs or features without clearing up these ambiguities. In the past, I’ve jumped into tasks only to hit a wall midway through because of an unaddressed detail. Having to reach out to a busy manager at that point breaks the implementation flow and is generally inefficient. For this meeting and these features, that proactive clarification was the key lesson.

- I ended up starting the Redis cache on startup feature first because it was more decoupled and independent from any of the existing business logic, which makes it easier for me to implement without having to think about how it affects existing logic and features. My initial thought process is to implement some sort of service that we can use to run not just this function, but any future scripts/functions on application startup. This is because we already have so many decoupled logic, event driven systems, and asynchronous programming in general, that it makes it hard to track where a particular code/domain executes and when. Having a service/module that says to developers `hey this is a service that is in charge of any type of scripts or functions on application startup` makes it easier for developer experience and maintainability in general.

- from my research, there was a couple of ways to go about this, and for spring in particular, there is a couple of ways to go about it, depending on where we want to execute the function based on the lifecycle of the application. There are ones that allow you to grab and run raw scripts, there is one that allows you to run before the application is considered "ready". However, the best one that I had found was that spring will fire an event called application ready event, which can be listend to by startup service. I researched a couple of the other events. Also that is sent automatically by spring throughout its lifecycle and it's pretty cool. There is around six where the application ready event is the last event that is sent when the application is fully started.

- one of the main concerns which I had almost overlooked during my research when using spring lifecycle application events is with horizontal scaling. And what that means is that if you have a Kubernetes pods which will spawn in two or more of the applications which is connected to the same redis server, it will essentially run the functions independently for each pod and if they're connected to the same Redis server, then it would be a race condition to see who will override the cache first. Fortunately I had clarified with my manager, and I also had some initial context, that our deployment is going to be isolated and on its own network. Meaning that we're not going to need Kubernetes and that the application is going to be the traditional server-client architecture where we have one server serving multiple clients, but it is enclosed within its own network due to business restrictions.

- For the initial implementation, I considered how the timing of lifecycle functions and our deployment architecture would affect the startup procedure. Since our initialization function relies on external services like Redis, InfluxDB, and MySQL, I researched the Spring `@Retryable` library. This library leverages Spring’s proxy system and Aspect-Oriented Programming (AOP). By annotating classes or functions with `@Retryable`, we can prevent the application from failing immediately if these external dependencies aren't alive yet. While I know we can manage service dependencies through Docker Compose health checks, I still wanted an application-level mechanism to ensure we have a retry window if a service is temporarily unavailable during or after startup.

- Researching `@Retryable` led me down a rabbit hole regarding Spring proxies and CGLIB. In a previous project, I had wondered whether the constructor of a proxied class would be called twice. I initially thought CGLIB might be unsafe for this reason, but I learned that modern Java applications use a library called Objenesis. This allows Spring to instantiate proxies without re-calling the constructor, making it safe to use.

- This also made me think about my previous work with asynchronous programming. I was concerned that if I implemented a retry mechanism, the main thread might wait for the retry interval and potentially freeze the application. Further research confirmed that while the thread executing the retry does wait, it doesn't freeze the entire app. I discovered that in a Spring Boot application, the main thread actually launches the Tomcat web server thread to manage the application lifecycle and then effectively "dies" or hands off control. This was a new lesson for me; I hadn't realized the main thread terminates in our specific architecture. Therefore, even if a startup function is synchronous and waits for a retry interval, it won't matter because the Tomcat threads are already active and handling the core business logic.

- After my research, I decided against using the `@Retryable` library. Upon reviewing the source, I realized it was essentially just a wrapper for retry mechanisms that felt like overkill for our needs. Adding an entire dependency for a single function would introduce unnecessary coupling and complexity. Instead, I implemented a custom retry mechanism using `Thread.sleep()`. Initially, I considered a simple service with a listener that directly called our business logic, but I realized we might need to add more startup tasks in the future. A basic `for` loop would have created a dependency issue where the retry count was shared across all tasks; for example, if the first task used two attempts, I would have to manually reset the counter for the second task. Implementing this with nested `if-else` blocks and boolean flags quickly became boilerplate-heavy.

  To solve this, I designed a solution using the Command Pattern. I created a lightweight Java `record` called `StartupTask` to store a task name and a `Runnable` function. This allows us to maintain a `List<StartupTask>` that the `onApplicationReady` method can iterate through. By passing each task into a dedicated `executeWithRetry` helper method, I ensured that each sequence has its own isolated retry counter and backoff timer. This architecture makes the system highly extensible; to add a new startup requirement, a developer just needs to define the function and add it to the list. This pattern of aggregating executors for an orchestrator—in this case, the `ApplicationReadyEvent` listener—is an approach I've successfully used in previous projects to manage complex initialization sequences.

````java
@Service
@Slf4j
public class SystemStartupService {

    // A lightweight Record to hold the task name and the method reference together
    private record StartupTask(String name, Runnable executable) {
    }

    /**
     * Listens for the server to be fully started and executes a sequence of startup actions.
     * This method is executed synchronously.
     */
    @EventListener(ApplicationReadyEvent.class)
    public void onApplicationReady() {
        log.info("=== Starting Application Startup Sequence ===");

        // Define your sequence of tasks here to be executed in order
        List<StartupTask> startupSequence = List.of(
                new StartupTask("Initial Data Synchronization", this::executeInitialDataSyncTask)
        );

        // Loop through the sequence
        for (StartupTask task : startupSequence) {
            boolean success = executeWithRetry(task.name(), task.executable());

            if (!success) {
                log.error("=== Application Startup Sequence stopped at Task: '{}' ===", task.name());
                return;
            }
        }
        log.info("=== Finished Application Startup Sequence Successfully ===");
    }

    /**
     * A reusable retry wrapper. It isolates the attempt counter and backoff timer
     * for whatever specific task is passed into it.
     * * @param taskName A descriptive name for the logs
     * @param task The actual method to execute (passed as a method reference)
     * @return true if successful, false if all retries were exhausted
     */
    private boolean executeWithRetry(String taskName, Runnable task) {
        int maxAttempts = 5;
        long backoffDelay = 5000; // 5-second initial wait

        for (int attempt = 1; attempt <= maxAttempts; attempt++) {
            try {
                log.info("Executing Task: '{}' (Attempt {}/{})", taskName, attempt, maxAttempts);
                task.run();
                log.info("Task '{}' completed successfully.", taskName);
                return true;
            } catch (Exception e) {
                log.warn("Task '{}' failed (Attempt {}/{}). Error: {}",
                        taskName, attempt, maxAttempts, e.getMessage());

                if (attempt == maxAttempts) {
                    log.error("CRITICAL: All {} retries exhausted for task '{}'.", maxAttempts, taskName);
                    return false;
                }

                try {
                    // Pause the main thread before trying again
                    Thread.sleep(backoffDelay);
                } catch (InterruptedException ie) {
                    Thread.currentThread().interrupt();
                    log.error("Retry sleep for task '{}' was interrupted.", taskName);
                    return false;
                }
            }
        }
        return false;
    }
